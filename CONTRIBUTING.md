# SpecLoom: The HADD Framework Engine

**SpecLoom** is the rigorous backbone for **Human-AI Hybrid Development**.

It is designed to be the antagonist of "vibe-coding." While Large Language Models (LLMs) are excellent at generating code that *looks* correct, they often struggle with long-term coherence, specific requirements, and non-functional constraints.

SpecLoom solves this by enforcing a strict **V-Model**, ensuring that every line of code generated by an AI (or human) is explicitly traceable to a validated design, which is traceable to a requirement, which is traceable to a stakeholder need.

---

## 🧠 Core Philosophy

### 1. AI-Native & Human-Readable
*   **For the AI:** The specification is stored as strict, schema-validated **JSON objects**. This ensures the AI Agent always has unambiguous, structured context (no "fuzzy" text).
*   **For the Human:** SpecLoom generates standard **IEEE-compliant documentation** (SRS, SDD) from these objects. Humans review readable docs; AI operates on structured data.

### 2. The 3-Phase V-Model
While we organize data into 8 directories for structure, the process follows three distinct phases:
1.  **Definition (The "Why" & "What"):** Defining Context, User Needs, Functional Specs, and **Architecture** (Stage 05).
2.  **Implementation (The "How"):** Writing code using strict **TDD** (Test-Driven Development).
3.  **V&V (The "Proof"):** Verification (Did we build it right?) and Validation (Did we build the right thing?).

See [Traceability Map](traceability_map.md) for the detailed dependency graph.

### 3. The "Guardian" of the Plan
"Plan-First" is not just a philosophy; it is the mechanism of control.
*   We do not code randomly.
*   We follow a **Task Graph** (`Stage 06`).
*   The Plan is **Adaptive**: It evolves. We plan the Spec creation. Once defined, we reshape the plan for Implementation. Once built, we reshape for V&V.

## 🏗️ Structure & Architecture

SpecLoom explicitly models Software Architecture (Views, APIs, ADRs) before coding begins.

| Directory | Content | Role |
| :--- | :--- | :--- |
| `01_context` | Stakeholders, Product Scope | **Definition** |
| `02_pivots` | Assumptions | **Definition** |
| `03_users` | User Requirements (UR) | **Definition** |
| `04_system` | Functional Reqs (FR), Business Rules | **Definition** |
| `05_design` | **Architecture Views**, APIs, Data Models | **Definition** |
| `06_execution` | **The Task Graph** | **Planning** |
| `src/` | **Implementation** | **Realization** |
| `tests/` | **Verification** | **V&V** |

## 🚀 The Guardian Workflow

SpecLoom acts as an interactive "Guardian" CLI that guides the AI/User through the lifecycle.

### 1. `loom status`
Visualizes the V-Model health and Plan progress.

```bash
$ loom status
--- Plan Status ---
Progress: 4% (1/24 Tasks)
  - Pending: 23

--- V-Model Artifacts ---
 [x]  Definition: Specification  (28 artifacts)
 [x]  Definition: Architecture   (11 artifacts)
 [x]  V&V: Verification          (1 artifacts)
```

### 2. `loom next` (The GPS)
Tells the agent exactly what to do next. It prevents "context drifting" by focusing on a single, atomic objective.

```bash
>>> NEXT OBJECTIVE: TASK-002 <<<
Title: Implement SpecNode Domain Entity
Type: Feature
TDD Cycle: Red -> Green -> Refactor
Instructions:
 [ ] Create test file tests/domain/SpecNode.test.ts
 [ ] Implement SpecNode class in src/core/domain/SpecNode.ts
```

### 3. TDD Integration
SpecLoom mandates TDD. A Feature Task is not "Done" until:
1.  A Test is created (Red).
2.  Implementation is written (Green).
3.  Verification confirms the pass.

## 🛠️ Usage

### Installation
```bash
npm install
npm run build
```

### MCP Server (AI Integration)
SpecLoom is built to run as a **Model Context Protocol (MCP)** server. Connect your IDE or Agent to it to give the AI "Project Awareness."

```json
{
  "mcpServers": {
    "specloom": {
      "command": "node",
      "args": ["/path/to/dist/src/adapters/mcp/server.js"]
    }
  }
}
```
